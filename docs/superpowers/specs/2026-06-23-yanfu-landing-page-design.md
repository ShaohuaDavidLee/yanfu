# 严复 Skill 官网设计规格

## 1. 文档状态

- **产品：** 严复 Skill
- **页面：** 严复 Skill 独立 Landing Page
- **版本：** v0.1
- **交互级别：** 静态产品页，保留必要的锚点导航、外链与下载入口
- **视觉方向：** 已确认采用“原文 / 译文”方案
- **视觉基准：** [yanfu-landing-selected-direction.png](assets/yanfu-landing-selected-direction.png)

本规格只定义严复官网。Skill 的输入、翻译流程和交付契约继续以
[2026-06-23-yanfu-v0.1-design.md](2026-06-23-yanfu-v0.1-design.md) 为准。

## 2. 页面目标

严复官网首先要让第一次访问的开发者在五秒内明白：

1. 严复处理的是已经存在的 Landing Page。
2. 它把开发者视角的表达翻译成用户能快速理解的表达。
3. 它不重新定位产品，不发明价值主张。
4. 用户需要提供公开 Landing Page 和 1–5 张核心产品截图。
5. 用户会得到新的 `landing.html` 和一份简短的严复译注。

页面的首要成功标准是准确理解与复述，不以夸张承诺或转化技巧制造点击。

## 3. 目标用户

### 主要用户

- 已经做出产品并拥有 Landing Page 的独立开发者。
- 能把功能做出来，但页面仍以技术、功能清单或实现细节为中心的开发者。
- 不希望轻易修改定位，只想先把现有产品讲清楚的创作者。

### 用户进入页面时的核心问题

- 严复到底改什么？
- 它会不会擅自改掉我的产品定位？
- 为什么新的页面会更容易理解？
- 我需要提供什么？
- 最后会得到什么？
- 我在哪里下载并使用这个 Skill？

## 4. 产品表达

### 品牌

**严复 Skill**

### 品牌说明

**你的产品故事翻译官**

### 首屏主标题

> 把你的「开发者视角」落地页，<br>
> 翻译成用户 5 秒能懂的落地页

桌面端允许分为两至三行，移动端允许分为三至四行。不得缩小到影响可读性，
也不得让“用户 5 秒能懂”脱离完整语义单独悬空。

### 首屏说明

> 不改产品定位，不发明价值主张。<br>
> 只忠于真实产品，重排故事结构与文案。

### 主要行动

- **主 CTA：** 查看翻译前后
- **次 CTA：** 下载 Skill

主 CTA 滚动到“原文 / 译文”对照区。次 CTA 滚动到“下载与使用”区，
不在首屏直接跳向尚未发布的地址。

## 5. 页面故事结构

页面使用下面的固定叙事顺序：

> Hero → 原文 / 译文 → 信・达・雅 → 翻译流程 → 翻译案例 → 下载与使用 → Skill 家族与开发者

该顺序的作用是先让用户看见结果，再解释原则和过程，最后提供使用入口。
不在证明严复价值之前展开长篇方法论。

## 6. 页面分区

### 6.1 顶部导航

左侧为品牌锁定：

- 严复头像或肖像资产。
- `严复 Skill`。
- 英文小字 `PRODUCT STORY TRANSLATOR`。

桌面端右侧依次展示：

- 包青天
- 司马迁
- 草诀歌
- 关于开发者
- 下载 Skill

前四项使用轻量文字链接，“下载 Skill”使用方角描边按钮。导航不得抢过主标题。

移动端保留品牌与“下载 Skill”按钮；四个外部入口移到首屏之后的横向链接行，
并在页尾完整重复。首版不增加汉堡菜单和抽屉。

### 6.2 Hero

Hero 使用选定视觉方向中的编辑式版面：

- 左侧或背景放低透明度严复线描肖像。
- 右上保留朱砂印章式视觉标记。
- 主标题是第一视觉层级。
- 品牌说明和正文均低于主标题。
- 使用一个实心主按钮和一个描边次按钮。

首屏不放表单、输入框、虚构用户评价或效果数据。

### 6.3 原文 / 译文

这是页面最重要的证明区，紧接 Hero。

桌面端采用三列：

1. **原文 SOURCE：** 开发者视角页面。
2. **严复译：** 朱砂色翻译箭头或校改符号。
3. **译文 TARGET：** 用户视角页面。

原文示例突出参数、规格、功能和技术术语。译文示例使用经典表达：

> 1000 首歌，装进口袋。

译文右侧增加四层翻译注解：

- 功能：它是什么。
- 动作：用户如何使用。
- 结果：它带来什么价值。
- 可复述表达：一句用户能记住并转述的话。

该示例只用于解释严复的方法，不暗示严复参与了 iPod 的真实传播工作。

移动端按“原文 → 严复译 → 译文 → 四层注解”纵向排列，不能并排压缩截图。

### 6.4 信・达・雅

该区用三条横向行而不是三张卡片：

- **信：忠于产品事实。** 只使用原页面、产品截图和开发者补充中可确认的证据。
- **达：让用户准确理解并能复述。** 五秒内明白产品是什么、为谁服务、有什么价值。
- **雅：结构清楚，表达克制。** 调整信息顺序与语言节奏，但不让修辞越过事实。

每一行包含一个朱砂单字、一个短标题和一段解释。三行之间只使用细分隔线。

### 6.5 严复的翻译流程

流程为四步：

1. **提交原文：** Landing Page 链接和 1–5 张核心产品截图。
2. **建立事实底稿：** 核对功能、动作、结果、证据和原价值主张。
3. **故事重排：** 重写信息层级、结构和文案，保留品牌与产品事实。
4. **交付译文与译注：** 输出 `landing.html` 和 `yanfu-notes.md`。

视觉上使用连续的校稿流程线、稿纸或文档轮廓，不使用四张悬浮 SaaS 卡片。

### 6.6 翻译案例

首版展示三个案例：

- Let’s Pod
- TransWeb
- 包青天.skill

每个案例均使用紧凑的“原文 / 译文”对照，包含：

- 产品名称和一句产品类别。
- 原页面缩略图。
- 翻译后页面缩略图。
- 一句翻译说明。

案例不写长篇产品诊断，不评价产品定位，也不显示未经验证的转化提升。

包青天案例名称同时链接到 `https://b.caojuege.com/`，作为案例原站入口。

### 6.7 下载与使用

该区回答三个问题：

- 需要提供什么。
- 会得到什么。
- 如何调用严复 Skill。

输入明确为：

- 一个公开可访问的 Landing Page URL。
- 1–5 张核心产品界面截图。

输出明确为：

- 响应式 `landing.html`。
- 简短的 `yanfu-notes.md`。

下载按钮使用 `下载严复 Skill`。在 Skill 尚未发布时：

- 本地预览中的按钮停留在本区，并显示“Skill 完成后开放下载”。
- 不链接到不存在的 GitHub 仓库，不伪造下载成功。

正式发布的部署门槛是配置一个可访问的 Skill 仓库或发行包 URL。未配置时不得上线
带有虚假下载行为的按钮。

### 6.8 Skill 家族与开发者

页尾深色墨底区域同时承担品牌归属和相关入口，不新增独立介绍页面。

#### Skill 家族

| 入口 | 显示名称 | URL | 说明 |
| --- | --- | --- | --- |
| 包青天 | 包青天.skill | `https://b.caojuege.com/` | 用证据判断需求 |
| 司马迁 | 司马迁.skill | `https://simaqian.caojuege.com/` | 为人作传与生成个人 OS |
| 草诀歌 | 草诀歌 AI Labs | `https://www.caojuege.com/` | 社区与作品入口 |
| 开发者 | 关于开发者 · 韶华 David | `https://www.caojuege.com/davidli` | 开发者个人主页 |

页尾品牌文案：

## 7. 视觉系统

### 7.1 与包青天的家族关系

官网沿用包青天.skill 的基础设计语言：

- 冷灰纸张背景。
- 黑色墨迹文字。
- 朱砂红印章与校改符号。
- 中文衬线标题、无衬线正文、等宽英文标签。
- 细灰分隔线。
- 方角按钮和克制的边框。
- 深色页尾。

严复不使用公堂、判词、案卷和判官语义。对应视觉隐喻改为：

- 原文与译文。
- 手稿与校样。
- 译注与批改。
- 稿纸、页码和校改符号。

### 7.2 基础 Token

实现时从包青天站点继承并收敛以下基础值：

- 页面背景：`#e5e5e5`
- 纸张表面：`#ececea`
- 主墨色：`#1a1a1a`
- 正文灰：`#5c5e60`
- 朱砂红：`#a8302d`
- 朱砂暗边：`#7d211e`
- 深色页尾：`#1a1a1a`
- 细分隔线：`rgba(71, 73, 75, 0.20)`
- 强分隔线：`rgba(71, 73, 75, 0.42)`

字体：

- 中文标题：`Noto Serif SC`
- 正文与界面：`Inter` 加系统中文无衬线回退
- 英文标签与元信息：`JetBrains Mono`

不使用渐变、圆角卡片、彩色光球、拟物纸张阴影堆叠或单一深蓝/紫色主题。

### 7.3 版心与密度

- 主内容版心约 `880px`。
- 原文 / 译文对照区可扩展到约 `1180px`，以保证截图可读。
- 桌面端左右安全边距不少于 `32px`。
- 移动端左右安全边距为 `18px`。
- Section 之间以留白和细线分隔，不把整段内容包进卡片。

## 8. 响应式规则

### 桌面端：`>= 960px`

- 页头显示完整四个外链与下载按钮。
- Hero 使用肖像、标题和印章构成不对称编辑式布局。
- 原文 / 译文使用三列。
- 案例可三列展示，但每列内部保持原文 / 译文对照可辨识。

### 平板端：`600px–959px`

- 页头只保留品牌、Skill 家族入口和下载按钮；家族入口允许压缩字距。
- 原文 / 译文改为两列，翻译箭头进入两列之间，四层注解放在译文下方。
- 案例按两列后单列排列。

### 移动端：`< 600px`

- 页头只保留品牌和下载按钮。
- 外部入口在 Hero 后以两行文字链接展示，并在页尾重复。
- 所有核心区块单列。
- 主标题不小于 `38px`，正文不小于 `15px`。
- 按钮可纵向排列，宽度稳定，不因文案变化发生布局跳动。
- 所有截图保持原始宽高比，不裁掉核心产品界面。

## 9. 交互与状态

首版只包含：

- 锚点平滑滚动。
- 页面内 CTA。
- 可访问的外部链接。
- 已发布后的 Skill 下载或仓库入口。

不包含：

- 在线提交 URL。
- 文件上传。
- 自动分析。
- 登录注册。
- 主题切换。
- 动画型案例轮播。
- 虚构的实时生成状态。

链接必须有清晰的 hover 和 focus 状态。用户使用键盘时，焦点顺序与页面阅读顺序一致。

## 10. 内容与资产契约

官网实现前需要准备：

- 严复肖像或可合法使用的线描资产。
- 朱砂印章资产或基于合法字体排版的简单印章组件。
- iPod 原文 / 译文方法示例。
- Let’s Pod、TransWeb、包青天三个案例的原页与译页截图。
- 严复 Skill 发布后的真实下载地址。

任何案例截图缺失时，删除该案例，不使用灰色占位框。任何效果数据没有来源时，
不得进入页面。

## 11. 技术结构

首版采用可直接托管的静态站点：

```text
landing/
├── index.html
├── assets/
│   ├── yanfu-portrait.*
│   ├── cases/
│   └── social-card.*
└── README.md
```

实现原则：

- HTML 使用语义化的 `header`、`main`、`section`、`nav` 和 `footer`。
- CSS 可内联在单页中，或拆为一个本地样式文件；不引入前端框架。
- JavaScript 只处理锚点、必要的导航状态和下载入口，不承载内容。
- 页面在禁用 JavaScript 时仍能阅读和访问外部链接。
- 外部 URL 集中定义，避免页头和页尾链接不一致。

## 12. 异常与降级

- Google Fonts 加载失败时使用系统字体回退，版面不能溢出。
- 肖像加载失败时隐藏背景肖像，标题与 CTA 仍保持正确位置。
- 案例截图加载失败时显示案例名称和文字说明，不显示破损图片图标。
- 下载地址未配置时显示明确的未发布状态，不执行空链接。
- 外部链接必须使用完整 HTTPS URL。

## 13. SEO 与分享

页面需要包含：

- `lang="zh-CN"`。
- 标题：`严复 Skill · 你的产品故事翻译官`。
- Description：`把开发者视角的 Landing Page，翻译成用户 5 秒能懂的 Landing Page。忠于产品事实，只重排故事结构与文案。`
- Canonical URL 在正式域名确定后由部署配置写入。
- Open Graph 标题、描述和一张真实页面分享图。

Canonical URL 未确定不阻塞本地实现，但正式部署前必须配置。

## 14. 验收标准

### 五秒理解测试

陌生用户浏览首屏五秒后，至少能准确说出：

- 这是一个改写 Landing Page 故事结构与文案的 Skill。
- 它服务于已经做出产品的开发者。
- 它不负责重新定位产品。

### 内容准确性

- 页面没有发明功能、指标、客户评价或转化效果。
- 输入与输出和 Skill v0.1 规格一致。
- iPod 示例被明确呈现为方法示例。
- 四个相关入口均使用本规格中的准确 URL。

### 视觉与响应式

- 视觉结果与已确认的“原文 / 译文”方向一致。
- 页面与包青天具有家族感，但没有公堂或判案语义。
- `1440px`、`768px`、`390px` 三种视口无横向溢出、文字遮挡和按钮截断。
- 原文 / 译文截图在移动端仍可辨认。

### 可用性

- 所有锚点和外链可用。
- 外链新标签页行为一致。
- 键盘可访问所有链接和按钮。
- Skill 未发布时不存在虚假下载。

## 15. 实施边界

本轮只实现严复官网，不同时开发严复 Skill。官网代码与 Skill 源码保持目录隔离。
严复 Skill 的开发在官网实现、案例素材与页面表达验证完成后另行进入实施计划。
