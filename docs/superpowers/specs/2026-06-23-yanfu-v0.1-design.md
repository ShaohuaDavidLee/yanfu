# 严复 Skill v0.1 设计规格

## 1. 产品定义

**产品名：** 严复 Skill  
**副标题：** 你的产品故事翻译官

严复把开发者已有的 Landing Page，翻译成用户更容易听懂的 Landing Page。

它不负责重新定位产品，不修改开发者的核心价值主张，也不做完整视觉重设计。它以原 Landing Page 和核心产品截图为证据，重新组织故事结构、文案和信息层级。

### 核心表达

> 严复，把开发者写给自己的 Landing Page，翻译成用户一眼能懂的 Landing Page。

### 翻译原则

- **信：** 忠于产品事实、开发者价值主张和真实能力。
- **达：** 把功能、流程和结果改写成用户能快速理解的语言。
- **雅：** 通过清晰的结构、节奏和克制的视觉调整增强表达，但不能以“雅”损害“信”。

## 2. v0.1 范围

### 包含

- 阅读一个公开可访问的 Landing Page。
- 接收 1–5 张核心产品界面截图。
- 提取原页面的价值主张、产品能力、用户结果、CTA 和品牌元素。
- 重构 Landing Page 的信息结构和文案。
- 沿用原页面的主要视觉语言。
- 删除重复、偏题、空泛或缺乏证据的内容。
- 输出可直接打开的响应式 HTML 页面。
- 输出一页左右的严复译注。

### 不包含

- 没有 Landing Page 的产品。
- 自行注册账号或登录后探索产品。
- 产品定位、目标市场或商业模式诊断。
- 竞品分析。
- 产品是否值得做的判断。
- 完整网站视觉重设计。
- 发明功能、效果数据、用户评价或客户案例。
- 一句话 Pitch、电梯推介、一分钟推介、15 分钟推介和 Slides 的独立生成模式。

后续版本可以在相同翻译底稿上扩展多种长度的 Pitch 和 Slides，但不进入 v0.1。

## 3. 输入契约

### 必填

1. Landing Page 链接。
2. 1–5 张核心产品界面截图。

### 可选

- 开发者最希望用户理解的一句话。
- 主要 CTA。
- Logo、字体或其他品牌素材。
- 不允许修改或删除的内容。

### 输入不足

如果页面无法访问、截图与产品无明显关系，或原页面无法确认基本价值主张，严复停止生成最终页面，并只请求最小必要补充材料。

严复不得用营销常识填补产品事实缺口。

## 4. 认知与翻译流程

### 4.1 阅读原页面

从 Landing Page 中提取：

- 开发者如何定义产品。
- 开发者声明的目标用户。
- 核心价值主张。
- 产品功能与使用方式。
- 用户结果。
- 数据、证明与客户评价。
- 主要 CTA。
- Logo、色彩、字体气质、组件样式和页面节奏。

### 4.2 阅读产品截图

从截图中只确认可见事实：

- 用户输入什么。
- 产品执行什么动作。
- 用户得到什么结果。
- 哪个界面最能证明产品价值。
- 页面中的功能承诺是否有真实产品界面支撑。

截图不能单独证明效率提升、准确率、用户满意度或其他效果指标。

### 4.3 建立内部翻译底稿

每项重要新表达必须记录：

| 字段 | 含义 |
| --- | --- |
| 新页面表达 | 严复准备使用的文案或结构 |
| 原始依据 | 原 Landing Page、截图或开发者补充 |
| 处理方式 | 保留、压缩、重组、弱化、删除 |
| 置信度 | 已确认、观察所得、需要确认 |

没有依据的表达不得进入最终 Landing Page。

### 4.4 重构故事

默认故事顺序：

> Hero → 用户处境 → 产品机制 → 核心界面 → 用户结果 → 证明 → FAQ → CTA

这不是强制模板。严复应根据产品事实删减或调整区块，不为凑齐结构而添加内容。

### 4.5 删除杂音

允许删除：

- 重复功能介绍。
- 与核心故事无关的区块。
- 空泛口号。
- 无证据的效果承诺。
- 过早出现的实现细节。
- 同义反复的 CTA。
- 妨碍理解的装饰内容。

不得悄悄删除开发者的重要承诺。关键删减必须在严复译注中说明。

## 5. 视觉边界

### 总原则

> 保留皮肤，重排叙事。

### 默认保留

- Logo 与品牌名称。
- 主色和字体气质。
- 按钮与组件风格。
- 图标、插图和品牌素材。
- 原有产品截图。
- 主要 CTA 及其真实链接。
- 整体视觉性格。

### 允许调整

- Hero 信息层级。
- Section 顺序。
- 标题、正文和 CTA 文案。
- 内容删减与合并。
- 截图选择、尺寸和出现位置。
- 留白、文字宽度和阅读节奏。
- 重复 CTA 的位置。

### 仅在妨碍理解时调整

- 对比度不足。
- 文字溢出或移动端错位。
- 装饰元素遮挡内容。
- 卡片嵌套造成层级混乱。
- 原页面结构无法响应式呈现。

## 6. 输出契约

### 6.1 `landing.html`

- 单文件、响应式、可直接打开。
- 使用原页面可合法获取的品牌素材和用户提供的产品截图。
- 主要链接与 CTA 可用。
- 不显示内部分析、置信度或咨询语言。
- 优先完成真实用户理解，不展示严复自身的方法论。

### 6.2 `yanfu-notes.md`

控制在一页左右。

#### 信

- 原始价值主张是什么。
- 忠实保留了什么。
- 重要新表达分别来自哪里。
- 哪些内容因缺乏证据没有使用。

#### 达

- 最终一句话 Pitch。
- 新页面的故事主线。
- 信息顺序为什么这样调整。
- 删除、合并或弱化了什么。

#### 雅

- 保留了哪些原设计元素。
- 为什么选择这些产品截图。
- 做了哪些必要且克制的视觉修正。

#### 待确认

- 需要开发者确认的表达。
- 没有足够证据进入页面的内容。

## 7. Skill 交互

### 触发示例

- “用严复翻译这个 Landing Page。”
- “这是网站链接和产品截图，帮我重新讲清楚。”
- “保持原来的设计，只重写结构和文案。”

### 开始时

严复检查是否同时具备链接和核心截图。

- 输入完整：直接开始。
- 只有链接：请求 1–5 张核心产品截图，不自行注册或登录。
- 只有截图：请求 Landing Page 链接。

### 完成时

返回：

1. 新 Landing Page 的本地预览链接。
2. 严复译注文件。
3. 一句话说明仍需开发者确认的内容。

## 8. Landing Page 官网

严复官网是一个独立产品页，信息密度和使用方式参考包青天.skill，但不复制其判案视觉。

### 页面结构

1. Hero：严复与核心一句话。
2. 信、达、雅：解释产品边界。
3. 输入：Landing Page 链接与核心截图。
4. 输出：新页面与严复译注。
5. 工作方式：读原文、看产品、重排故事、交付译文。
6. 三个案例。
7. 下载 Skill 与使用方法。
8. 开源仓库与草诀歌 AI Labs。

### 案例

#### Let’s Pod

展示如何把远程播客录音功能，翻译成用户能立即理解的核心使用结果。

#### TransWeb

展示如何把版本管理和 AI 内容编辑功能，翻译成一个具体、可感知的使用场景。

#### 包青天.skill

展示严复也能处理 Skill 型产品：保留产品人格和方法论，同时把输入、输出与用户价值讲清楚。

案例应优先展示原页面与严复译文页面的对照，而不是长篇评价。

## 9. 首版成功标准

对每个案例分别邀请开发者与陌生用户验证：

1. 开发者认为新页面忠于原产品。
2. 陌生用户在五秒内能说出产品是什么。
3. 陌生用户能说出产品带来的核心结果。
4. 开发者愿意采用新页面或其中大部分结构与文案。

首版不以真实转化率作为唯一成败标准。

## 10. 工程交付

首版包含两个独立部分：

1. 可安装的 `yanfu` Skill。
2. 类似包青天.skill的独立 Landing Page。

Skill 源码与官网源码放在同一个项目中，但保持独立目录，避免官网实现细节进入 Skill 上下文。
